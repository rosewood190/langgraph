# 竞赛模式实现总结

## 实现概述

根据用户反馈，重新设计了竞赛模式的实现方式，采用更简洁的方案：

### 核心改进

1. **启动时引导选择模式**：交互模式下，agent 会引导用户选择教学或竞赛模式
2. **只保留两种模式**：移除面试模式，只保留 teaching 和 contest
3. **模式只影响表述**：不改变工作流程，只调整各 agent 的表述风格
4. **移除双代码策略**：竞赛模式直接生成 LeetCode 格式，跳过在线验证
5. **竞赛模式默认 C++**：如果用户不指定语言，默认使用 C++

## 技术方案

### 模式选择流程

```
启动交互模式
    ↓
显示模式选择提示
    ↓
用户输入 1 或 2
    ↓
设置 mode = "teaching" 或 "contest"
    ↓
开始对话
```

### 模式影响范围

```
mode 参数传递到各 agent
    ↓
analyst: 调整题意分析表述
planner: 调整策略讲解表述
pseudocode: 调整伪代码讲解表述
coder: 选择代码格式（标准I/O vs LeetCode）
code_runner: 决定是否跳过验证
formatter: 调整最终输出风格
```

### 竞赛模式特点

1. **代码格式**：LeetCode 标准类方法
   - C++: `class Solution { public: ... };`
   - Python: `class Solution: def ...`
   - Java: `class Solution { ... }`

2. **跳过验证**：由于类方法格式无法直接运行，跳过 Wandbox 验证

3. **默认语言**：C++（符合竞赛习惯）

4. **表述风格**：
   - 题意分析：直接陈述要求
   - 策略规划：简洁说明算法选择
   - 复杂度：使用"复杂度分析："格式
   - 验证结果：使用 ✓/✗ 符号

## 使用示例

### 交互模式

```bash
$ python -m app.main

你好，我是你的算法设计多智能体助手。
我支持两种输出模式：
  1. 教学模式 - 口语化表述，适合学习理解
  2. 竞赛模式 - 专业严谨表述，生成 LeetCode 标准格式代码

请选择你想使用的模式（输入 1 或 2，默认为教学模式）：
2

已选择竞赛模式。
我会像老师一样分阶段陪你做题：先拆题，再讲策略，最后再落到实现。
每一步讲完我都会停下来等你反馈，你可以让我继续、重讲，或者直接切换到代码。输入 exit 可退出对话。

User> 给定一个整数数组 nums 和一个目标值 target，找出和为目标值的两个整数的下标。
```

### 命令行模式

```bash
# 使用竞赛模式
python -m app.main --mode contest "两数之和问题"

# 使用教学模式（默认）
python -m app.main "两数之和问题"
```

## 优势总结

### 相比之前的双代码策略

1. **更简洁**：不需要生成两个版本的代码
2. **更清晰**：模式只影响表述，不改变工作流
3. **更直接**：竞赛模式直接生成 LeetCode 格式
4. **更高效**：减少 token 消耗和生成时间

### 用户体验改进

1. **启动引导**：交互模式下主动引导用户选择
2. **模式明确**：只有两种模式，选择更简单
3. **符合习惯**：竞赛模式默认 C++，符合竞赛习惯
4. **表述一致**：同一模式下所有 agent 表述风格统一

## 总结

新的实现方案更加简洁和实用：

1. **启动引导**：用户可以在交互模式下选择模式
2. **表述调整**：模式只影响各 agent 的表述风格
3. **代码格式**：竞赛模式生成 LeetCode 格式，教学模式生成标准 I/O
4. **跳过验证**：竞赛模式跳过在线验证（LeetCode 格式无法运行）
5. **默认 C++**：竞赛模式默认使用 C++

这个方案完全满足了用户的需求，同时保持了代码的简洁性和可维护性。
